## Recent Changes (2026-06-28)

### 1. FIFA R32 Bracket Structure
**File:** `model/monte-carlo.js` — `buildKOBracket()`

The bracket array was completely rewritten to match the official FIFA schedule from Wikipedia:
- R32 bracket ordered so sequential pairing produces correct R16 cross-pairing
- R16 pairings verified 8/8 correct against FIFA official bracket
- **16/16 R32 matches verified correct** against actual results
- Source: https://zh.wikipedia.org/wiki/2026年國際足協世界盃

**R32 bracket order (bracket position → FIFA match):**
```
[0]=M74  [1]=M77  → R16 M89: Germany/Paraguay vs France/Sweden
[2]=M73  [3]=M75  → R16 M90: S.Africa/Canada vs Netherlands/Morocco
[4]=M76  [5]=M78  → R16 M91: Brazil/Japan vs Ivory Coast/Norway
[6]=M79  [7]=M80  → R16 M92: Mexico/Ecuador vs England/DR Congo
[8]=M83  [9]=M84  → R16 M93: Portugal/Croatia vs Spain/Austria
[10]=M81 [11]=M82 → R16 M94: USA/Bosnia vs Belgium/Senegal
[12]=M86 [13]=M88 → R16 M95: Argentina/Cape Verde vs Australia/Egypt
[14]=M85 [15]=M87 → R16 M96: Switzerland/Algeria vs Colombia/Ghana
```

**Key fixes applied:**
- M83: Changed from `W('K') vs R('L')` to `R('K') vs R('L')` (Portugal is R(K), not W(K))
- M88: Changed from `R('D') vs 3rd(G)` to `R('D') vs R('G')` (Egypt is R(G), not 3rd(G))

**Note:** The simulation uses model predictions for group stage, so the specific teams may differ from actual FIFA results. The bracket STRUCTURE is correct; teams depend on the data loaded in localStorage.

### 2. Third-Place Assignment
**File:** `model/monte-carlo.js` — `buildKOBracket()`

Uses backtracking with slot constraints to assign 8 third-place teams to 8 winner slots:
- Each slot has `eligible` groups (excluding self and R16 opponent)
- Adjacent match's third-place team is also excluded
- Greedy fallback for edge cases

**Known limitation:** When the simulation's predicted third-place teams differ from actual FIFA results, the bracket may show different matchups. This is inherent to the model's prediction accuracy.

### 3. R16 Cross-Pairing

**File:** `compute-server.js` + `mc-worker.js`

Monte Carlo simulation now uses `worker_threads` for parallel execution:
- Up to 16 workers (CPU cores - 1)
- Small N (≤2000) runs single-threaded to avoid worker overhead
- 50,000 simulations: ~3s (vs ~10s single-threaded)

## Known Issues

### 0. API Data Source May Not Match Actual FIFA Results (HIGH)
The `/api/results-alt` endpoint returns data that may not match actual FIFA results. For example, Group K results show Colombia winning with 7pts, but the actual FIFA bracket shows Portugal as Group K winner.

**Impact:** If the API data is incorrect, the bracket will show wrong teams even though the bracket structure is correct.

**Mitigation:** Users should verify that the data in localStorage matches actual FIFA results. A `clear.html` utility page is provided to clear localStorage and re-fetch data.

### 1. Third-Place Assignment Mismatch (MEDIUM)
When the simulation predicts different group standings than actual results, the bracket shows different third-place teams. Example:
- Simulation: Iran (3rd in G) → bracket slot
- Actual FIFA: Bosnia Herzegovina (3rd in B) → bracket slot

**Root cause:** The bracket construction uses simulation-predicted standings, not actual FIFA results. This is inherent to the model's prediction accuracy.

**Possible fix:** Use actual results for bracket construction when available (requires passing actual group standings to `buildKOBracket`).

### 2. R16 Cross-Pairing Dependency (LOW)
The R16 cross-pairing uses hardcoded bracket indices. If the bracket array order changes, the R16 pairing will break.

**Mitigation:** The bracket array order is now fixed to match FIFA schedule. Any changes require updating both the bracket array and the R16 pairing.

### 3. Date Sorting Edge Case (LOW)
The date sorting uses zero-padded comparison: `"07/01"` vs `"07/10"`. This works correctly but requires the padding function to be applied consistently.

### 4. Third-Place Eligibility Constraints (LOW)
The backtracking algorithm for third-place assignment uses `eligible` arrays that exclude groups based on R16 opponent proximity. These constraints may not cover all edge cases for unusual group stage outcomes.

**Mitigation:** Greedy fallback handles cases where backtracking fails.

### 5. Compute Server Single-Threaded (INFO)
The compute server runs on a single Node.js thread. Heavy Monte Carlo simulations (50,000+ iterations) may block other requests.

**Current mitigation:** Simulations run on-demand, not continuously.

## API Endpoints

### server.py (port 9090)
| Endpoint | Method | Description |
|----------|--------|-------------|
| `/` | GET | Serve index.html |
| `/api/results-alt` | GET | Proxy to football-data.org (no key needed) |
| `/api/results` | GET | Proxy to football-data.org (needs API key) |
| `/api/odds` | GET | Proxy to the-odds-api.com |
| `/api/compute` | POST | Proxy to compute-server.js |

### compute-server.js (port 9091)
| Endpoint | Method | Description |
|----------|--------|-------------|
| `/health` | GET | Health check |
| `/compute` | POST | Computation actions |

**Actions:**
| Action | Description |
|--------|-------------|
| `simulation` | Full group stage + knockout simulation |
| `reevaluate` | Recompute probabilities for existing results |
| `calibration` | Calibration statistics |
| `caldiag` | Calibration diagnostics (reliability diagram + optimal T) |
| `ev` | Expected value analysis |
| `brier` | Brier score computation |
| `backtest` | Backtest on historical data |
| `montecarlo` | Monte Carlo simulation (N iterations, parallel via workers) |
| `full` | All of the above combined |

**Rate limiting:** 20 requests per 10-second window. Returns 429 with `{error, type: 'RATE_LIMIT', retryAfter}`.

**Structured errors:** All errors return `{error, type, action, suggestion}`.

**Request logging:** Each request logs action, elapsed time, and result size.

## Model Pipeline

### Group Stage Prediction
```
1. getFormAdjustedLambdas(home, away, formMap, date, marketProbs)
   → [lambda_home, lambda_away]
2. getBlendedProbs(home, away, formMap, date, marketProbs)
   → {win, draw, loss} probabilities
   Steps: Dixon-Coles → GBDT blend (20%) → Temperature scaling (auto-optimized T)
```

### Knockout Match Prediction
```
1. getFormAdjustedLambdas(home, away, formMap, date, marketProbs)
   → [lambda_home, lambda_away]
2. detScore(home, away, lh, la, kodate)
   → {ga90, gb90, ga, gb, method, lh, la, top3}
   Method: Seeded Poisson sampling (deterministic per match)
   Extra time: ET_FATIGUE=0.85 decay factor
   If draw: Elo-based penalty winner
   top3: [{s:"1-1", p:0.14}, {s:"2-1", p:0.09}, ...] — confidence display
```

### Monte Carlo Simulation (Parallel)
```
1. If N ≤ 2000: single-threaded
2. If N > 2000: spawn min(16, cpus-1) workers via worker_threads
3. Each worker runs batch independently
4. Results merged: count/N = probability
5. Returns: champ, finalist, semi, quarter, r16, simulationHistory
```
